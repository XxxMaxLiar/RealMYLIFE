**Канареечный деплой (Canary Deployment)** — это стратегия развертывания, при которой новая версия приложения внедряется постепенно, сначала на небольшой части трафика или серверов (напр., 5%, 10%, 25%).

"канарейкой" является небольшая группа подов (или серверов) с новой версией. Если с ними что-то не так (возрастает ошибка 5xx, падает производительность), это служит ранним сигналом отменить релиз, прежде чем он затронет всех пользователей.

Вот пошаговый план:
1. **В пайплайне после успешного прохождения всех тестов и сборки образа джоб деплоя (`deploy-to-prod`) ожидал **ручного подтверждения** (`when: manual`). Это была «кнопка» для ответственного, который давал отмашку на запуск канарейки.
	У меня был не статический файл `inventory.ini`, а **динамический скрипт инвентаризации** на Python или просто скрипт, Что делал этот скрипт:
    
    - Он подключался к API Kubernetes (`kubectl get pods -l app=myapp -o json`).
        
    - Анализировал, какие поды сейчас запущены с какими версиями (версия определялась по label'у, например, `version: {{ ansible_tag }}`).
        
    - **Формировал две группы хостов динамически:**
        
        - Группа `canary`: в неё попадало небольшое подмножество подов (скажем, 1-2), которые нужно обновить.
        - Группа `stable`: все остальные поды, которые пока остаются на старой версии.
            
2. **Само выполнение плейбука:**
    - В плейбуке было два разных play, нацеленных на эти группы.
    - **Первый шаг (Canary):**
        yaml
        - name: Deploy canary version to a subset of pods
          hosts: canary  # <-- Динамическая группа из 1-2 подов
          tasks:
            - name: Update image version on canary pods
              kubernetes.core.k8s:
                state: present
                definition: "{{ lookup('template', 'deployment.yml.j2') }}"
	Здесь я приостанавливал выполнение на 5-10 минут
          - name: Pause to monitor canary
            pause:
              minutes: 5
        
Во время паузы я и моя команда вручную проверяли дашборды в **Grafana** (ошибки 5xx, время ответа). Если за это время не приходило критичных алертов — всё было хорошо.
    - **Второй шаг (Full Rollout):**
        yaml
        - name: Deploy new version to all pods
          hosts: stable  # <-- Динамическая группа со всеми оставшимися подами
          tasks:
            - name: Update image version on all pods
              kubernetes.core.k8s:
                state: present
                definition: "{{ lookup('template', 'deployment.yml.j2') }}"
          # Уже без паузы, так как канарейка успешно прошла проверку
        

1. **Откат (Rollback):** Если на этапе паузы мы видели проблемы, процесс просто **прерывался**. Поскольку основная группа `stable` не была тронута, она продолжала стабильно работать. Канареечные поды можно было быстро вернуть к старой версии тем же Ansible, но обычно хватало простого удаления этих подов — тогда Deployment автоматически создавал их с предыдущей, стабильной версией из-за механизма реплик.
