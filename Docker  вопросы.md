
#### 1. **Что такое Docker и зачем он нужен?**
Docker — это платформа для создания, доставки и запуска контейнеризированных приложений.
- Контейнеры изолируют приложение и его зависимости.
- **Кластер** — группа серверов (нод), работающих как единая система.
- **Оркестратор** — инструмент для управления кластером (Kubernetes, Docker Swarm, Nomad).
- **Docker-контейнер** — это изолированная среда для запуска приложений, которая включает в себя: код, библиотеки, какой то requirements. Контейнеры работают поверх ядра операционной системы (ОС), но не содержат собственной ОС.
#### 2. **Чем контейнеры отличаются от виртуальных машин?**
- **Контейнеры** используют общий хост-ядро ОС, занимают меньше ресурсов и запускаются быстрее.
- **Виртуальные машины** эмулируют полный хост, что требует больше памяти и CPU.
#### 3. **Что такое Docker Image?**
Docker Image — это шаблон для создания контейнеров.
- Содержит всё необходимое для запуска приложения: код, зависимости, системные утилиты.
- Создается с помощью `Dockerfile`.

#### 4. **Какой процесс создания и запуска контейнера?**
1. Написать `Dockerfile`.
2. Собрать образ через `docker build`.
3. Запустить контейнер: через `docker run.
#### 5. **Что такое `Dockerfile` и какие его ключевые инструкции?**
`Dockerfile` — это текстовый файл с инструкциями для создания Docker Image.
- Основные инструкции:
    - `FROM` — базовый образ.
    - `RUN` — выполнение команд в образе.
    - `COPY` — копирование файлов в образ.
    - `CMD` — точка входа в контейнер.
#### 6. **Как посмотреть работающие контейнеры и остановить их?**
- Просмотр: `docker ps` 
- Остановка: `docker stop.
#### 7. **Что такое Docker Compose?**
Docker Compose используется для управления несколькими контейнерами на одном хосте.
- Конфигурация через `docker-compose.yml`.
- Команда запуска: `docker-compose up`.
#### 8. **Как уменьшить размер Docker Image?**
- Объединять команды `RUN` в одну строку.
- Удалять временные файлы (команда `--no-cache` и `rm`).
#### Docker sworm
**Docker Swarm** — это встроенная система оркестрации контейнеров, предоставляемая компанией Docker. Она позволяет масштабировать кластер из Docker-хостов, превращая их в единый логический хост, на котором можно запускать контейнеры.
#### 9. Чем COPY отличается от ADD?
COPY просто переносит с места сборки файлы, а ADD позволяет файлы выгружать из интернета и также с помощью ADD можно распаковывать архивы. ADD может переносить место сборки, в том смысле, что она может подтягивать данные не только из build контекста.
То есть **build context** — это директория, из которой запускается сборка.
Команда `COPY` может копировать файлы **только из build context** внутрь образа. а ADD в свою очередь может фактически привносить данные извне build contextа. Именно в этом смысле можно сказать, что `ADD` «переносит место сборки», потому что она расширяет источник данных за пределы локального контекста. 
**Почему это не всегда хорошо**:  `ADD` ухудшает воспроизводимость сборки, ломает кэш при изменении удалённого ресурса, создаёт проблемы в CI без интернет-доступа. Поэтому лучшая практика: это использовать `COPY` по умолчанию, но использовать `ADD` только если действительно нужна распаковка.
#### 10. Чем CMD отличается от ENTRY POINT
- **`ENTRYPOINT`** — это **неизменяемая программа**.
- **`CMD`** — это **аргументы по умолчанию** для этой программы. Если мы передаём свои аргументы при запуске контейнера (`docker run`), они **перезаписывают `CMD`**, но не `ENTRYPOINT`. 
- `ENTRYPOINT` можно перезаписать через `--entrypoint`, В этом случае Docker полностью игнорирует `ENTRYPOINT`, указанный в Dockerfile, и использует тот, который передан в команде запуска.

#### 11. Почему плохо перечислять несколько команд в команде run?
Docker изначально спроектирован по принципу: **один контейнер — один основной процесс**.
Если в одном контейнере запускаются: приложение, cron, nginx, миграции и ещё какие-то фоновые процессы, то становится сложно масштабировать компоненты отдельно, сложно мониторить и перезапускать. также если одна команда падает, то сложно понять какая конкретно команда вызвала проблему. Лучше проектировать контейнер так, чтобы он запускал один основной процесс, а дополнительные задачи выносить в отдельные контейнеры.

#### 12. Где лучше располагать наиболее изменяемые слои в dockerfile?
В Docker каждый слой кэшируется. Если меняется инструкция в Dockerfile, то пересобирается этот слой и все последующие слои. Предыдущие слои берутся из кэша, если их содержимое не изменилось. поэтому я буду располагать инструкции в Dockerfile от максимально стабильных к максимально часто изменяемым, потому что Docker идёт сверху вниз - сначала он берёт инструкцию, Сравнивает её с предыдущей сборкой и Если слой изменился — кэш для него и всех нижележащих слоёв сбрасывается.

#### 13. А каждая новая команда это всегда новый слой или нет? Есть ли какие-то команды, которые не создают новых слоев при сборке образа?
Слой создаётся, когда инструкция изменяет файловую систему образа это `RUN` , `COPY`,  `ADD`. 
Есть инструкции, которые **не создают слой**, потому что они не меняют файловую систему, а только добавляют метаданные. Это: `CMD`, `ENTRYPOINT`, `ENV`, `EXPOSE`, `WORKDIR`.

#### 14. В каких режимах сети мы можем запустить контейнер и какая из сетей используется по умолчанию?
Контейнер можно запустить в режимах **bridge** - `Это изолированная сеть, где контейнеры могут общаться друг с другом если находятся в одной сети, а наружу выходят через NAT`, **host** - `когда контейнер использует сетевой стек хоста напрямую. У него нет собственного IP — он работает на сетевом интерфейсе машины`, **none** - `когда Контейнер полностью изолирован от сети.` и **container** - `контаинер может использовать сетевой namespace другого контейнера`, а по умолчанию используется bridge.
